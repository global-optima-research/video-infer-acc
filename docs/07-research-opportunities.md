# 研究机会分析

> 调研日期：2026-03-06

基于对视频模型推理加速 6 大方向的全面调研，识别出以下研究空白和机会。

---

## 机会总览

| 优先级 | 方向 | 新颖性 | 工程量 | 潜在影响 |
|--------|------|--------|--------|----------|
| **1** | 少步模型缓存失效 + 协同设计 | 高 | 中 | 直接出论文 |
| **2** | DuCa 随机性理论推广 | 很高 | 低 | 理论贡献大 |
| **3** | 长视频专项加速 | 高 | 中 | 实际需求强 |
| **4** | 自适应组合框架 | 中 | 高 | 工程价值大 |
| **5** | 统一 benchmark | 中 | 高 | 社区价值大 |
| **6** | 消费级 GPU 端到端优化 | 低 | 中 | 用户群大 |

## 1. 少步模型的缓存失效问题 ★ 最推荐

### 现状

- 步数蒸馏（DCM/DOLLAR）将 50 步压缩到 4 步，每步信息密度大幅增加
- TeaCache/AdaCache 等缓存方法为 50 步设计（相邻步差异小，>97% 相似）
- 4 步模型中每步都很关键，步间相似性假设可能不成立

### 研究问题

1. 蒸馏后的 4 步模型上，各缓存方法是否还有效？衰减多少？
2. 4 步模型中 block 级 / token 级缓存是否比 step 级缓存更合适？
3. DCM+TeaCache 报告 54x 加速，但这个组合是否最优？
4. 能否在蒸馏过程中引入缓存感知（cache-aware distillation）？

### 可能的贡献

- 首个系统研究蒸馏 × 缓存交互效应的工作
- 设计专门针对少步模型的缓存策略
- cache-aware distillation：蒸馏时优化 student 的步间特征相似性，使其对缓存更友好

### 实验方案

```
基线模型：Wan2.1-14B (50步)
蒸馏模型：DCM 4步 / DOLLAR 4步

实验 1：在 4 步模型上测试所有缓存方法
  - TeaCache / AdaCache / DuCa / MixCache / EOC
  - 对比 50 步模型上的加速比和质量衰减

实验 2：粒度对比
  - Step 级缓存 vs Block 级缓存 vs Token 级缓存
  - 在 4 步 vs 50 步模型上的效果差异

实验 3：协同设计
  - 蒸馏 loss 中加入步间特征相似性正则项
  - 缓存策略参与蒸馏过程的联合训练
```

## 2. DuCa "随机 ≈ 重要性" 的理论推广

### 现状

DuCa 发现：在 token 缓存中，**随机选择 token ≈ 按注意力分数选择 token**，且随机选择天然保证多样性，还兼容 FlashAttention。

这是一个深刻但未被充分探索的 insight。

### 研究问题

1. 这个现象是否普遍？在以下场景中是否同样成立：
   - 随机选择哪些 block 缓存 vs 按相似度选择？
   - 随机选择哪些 layer 量化 vs 按敏感度分析选择？
   - 随机选择蒸馏的中间步骤 vs 精心设计的 schedule？
2. 如果"随机"普遍有效，说明什么？
   - Diffusion 模型内部是否存在**均匀分布的信息冗余**？
   - 这是否与 diffusion 模型的去噪过程（逐步去除均匀噪声）有关？
3. 随机选择的理论最优比例是多少？与模型规模/步数/分辨率的关系？

### 可能的贡献

- 建立"随机选择在扩散模型中普遍近似最优"的理论解释
- 信息论视角：量化 diffusion 模型中冗余的分布均匀性
- 实践指导：如果随机选择足够好，则无需复杂的选择策略，大幅简化方法设计

### 实验方案

```
实验 1：跨粒度验证
  - Block 选择：随机 vs 相似度 vs 梯度
  - Token 选择：随机 vs attention vs L2-norm
  - Step 选择：随机跳过 vs TeaCache 自适应
  在 3+ 模型上验证

实验 2：信息论分析
  - 计算各 block/token/step 的互信息
  - 验证是否均匀分布

实验 3：最优随机比例
  - 扫描 10%-90% 随机比例
  - 在不同模型规模和步数下的 Pareto 前沿
```

## 3. 长视频专项加速

### 现状

- 几乎所有方法在 2-5 秒短视频上验证（16-81 帧）
- 实际需求是 30s-2min（720-3600 帧）
- 长视频的帧间冗余远高于短视频（场景内 cosine similarity >95%）
- 现有缓存只利用了**步间冗余**，没有充分利用**帧间场景级冗余**

### 研究问题

1. 长视频中场景级冗余如何量化？场景切换点处冗余如何变化？
2. 能否设计场景感知的缓存策略？场景内极激进缓存，场景切换时全量计算
3. 分段并行 + 段间特征共享是否可行？
4. 渐进式生成（先关键帧，再插值）vs 全序列生成的质量-速度权衡？

### 可能的贡献

- 首个系统研究长视频加速的工作（>30 秒）
- 场景感知缓存策略
- 长视频特有的帧间冗余利用方法

## 4. 自适应组合框架

### 现状

- 6 个加速方向正交可叠加（理论 108x，实测 50-200x）
- 但所有组合都是**手动堆叠、静态配置**
- TurboDiffusion（200x）是当前最好的组合，但它是固定配置
- AdaCache 的"运动感知"和 MixCache 的"三粒度自适应"证明了自适应思路可行

### 研究问题

1. 给定输入 prompt + 硬件约束 + 质量要求，如何自动选择最优方法组合？
2. 能否用轻量预测模型（prompt → 最优配置）替代人工调参？
3. 各方法之间是否存在负交互（叠加后 1+1<2）？如何检测和规避？

### 可能的贡献

- 自适应加速配置选择器
- 方法交互效应的系统研究
- 类似 AutoML 的自动加速配置搜索

### 设想

```
输入: prompt + target_quality + hardware + latency_budget
  │
  ▼
[轻量分析器] → prompt 复杂度 / 运动强度估计
  │
  ▼
[配置选择器] → 从预定义配置空间中选择最优组合
  │
  ├── 简单场景 → 激进缓存 (TeaCache aggressive) + 少步 + 高量化
  ├── 复杂场景 → 保守缓存 + 更多步数 + 低量化
  └── 低端 GPU → 自动启用 offloading + 更激进量化 + VAE tiling
  │
  ▼
输出: 最优配置 + 预估质量/速度
```

## 5. 统一 benchmark

### 现状

- 每篇论文用不同的模型、分辨率、帧数、评测指标
- 无法公平对比不同方向的方法
- FastGen 有蒸馏 benchmark，FastVideo 有推理 benchmark，但没有跨方向的统一基准

### 研究问题

1. 如何设计涵盖所有 6 个方向的统一评测框架？
2. 质量指标如何兼顾感知质量和时间一致性？
3. 如何评测方法的叠加效果？

### 可能的贡献

```
VideoAccelBench:
├── 模型固定: Wan2.1-14B + HunyuanVideo
├── 分辨率/帧数: 480p/720p, 81/121 帧
├── 评测维度:
│   ├── 速度: 端到端延迟 + DiT-only 延迟 + 首帧延迟
│   ├── 质量: VBench + FVD + 人工偏好
│   ├── 资源: 峰值显存 + 平均 GPU 利用率
│   └── 鲁棒性: 100 条不同复杂度 prompt 的方差
├── 方法覆盖: 6 个方向 × 各方向 top-3 方法
└── 组合测试: 推荐组合的叠加效果
```

## 6. 消费级 GPU 端到端优化

### 现状

- RTX 4090 是性价比最优选择（$0.005/视频 vs 云 API $0.15-$2.50）
- 但大部分论文在 H100/A100 上做实验
- 24GB VRAM 是硬约束，需要不同的优化策略

### 研究问题

1. 4090 上各方法的最优组合是什么？（与 H100 可能不同）
2. VRAM 受限下，offloading vs 量化 vs tiling 的最优权衡？
3. 面向 5090 (32GB) 和未来 RTX 6090 的前瞻优化？

### 可能的贡献

- 消费级 GPU 专项加速指南
- 一键部署工具（类似 llama.cpp 对 LLM 的意义）
- VRAM 感知的自动配置

## 交叉机会

```
机会 1 (缓存×蒸馏) ←──互补──→ 机会 2 (随机性理论)
       │                              │
       │  理论 2 可解释为什么          │
       │  缓存在少步模型中             │
       │  失效或仍然有效               │
       │                              │
       ▼                              ▼
机会 4 (自适应框架) ←──需要──→ 机会 5 (统一 benchmark)
       │                              │
       │  框架需要 benchmark            │
       │  来评估配置效果               │
       │                              │
       ▼                              ▼
机会 3 (长视频) ←──应用场景──→ 机会 6 (消费级 GPU)
```

**最高效路径**：从机会 1（缓存×蒸馏）或机会 2（随机性理论）入手，工程量小，学术新颖性高，且能为后续机会 4/5 提供基础。
